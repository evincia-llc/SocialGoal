# Trigger unknown: CLOSED (Sprint 2, 2026-07-24)

The LMRR carried "database triggers unknown" as an open risk: hidden logic in
triggers would not appear in any code read and could silently diverge behavior
after the EF Core migration.

**Answer: there are no triggers, and by construction there cannot be any.**

Two legs, either sufficient alone:

1. **By construction (decision D1, 2026-07-23).** There is no live deployment,
   no production database, and no DBA-managed instance. Every SocialGoal
   database that exists is generated by EF6 Code First from the model in
   `SocialGoal.Data`, and nothing in the codebase or its deployment artifacts
   creates a trigger (no source, config, or SQL file contains a `CREATE
   TRIGGER` statement; this document is the string's only occurrence).
   A trigger would have to be added by hand to a database no one operates.

2. **Empirically.** A database was built from the committed baseline DDL
   (`schema-baseline.sql`) on `(localdb)\MSSQLLocalDB` and queried on
   2026-07-24:

   | Query | Result |
   |---|---|
   | `SELECT COUNT(*) FROM sys.triggers` | 0 |
   | `SELECT COUNT(*) FROM sys.server_triggers` | 0 |
   | `SELECT COUNT(*) FROM sys.tables` | 30 (matches the baseline) |

   The characterization test databases EF generates during CI runs are built
   the same way from the same model.

Consequence for the migration: no trigger-resident behavior needs porting or
characterization; the EF Core schema-parity work (Sprints 6-7) can treat the
baseline DDL as the complete database behavior surface.

Methodology note (already in `ai-context/lmrr-feedback.md`): when an engagement
confirms "no live database" the LMRR template could close this unknown by
construction instead of carrying it open.
